**Question 1:**
A TensorRT-LLM deployment of a 13B parameter LLM on Triton Inference Server is experiencing a consistent throughput degradation of 60% under sustained load of 200 concurrent users, compared to benchmarks at 10 concurrent users. GPU utilization is at 100%. Memory utilization is 91%. Latency per token increases from 18ms at low load to 74ms under peak load. What is the MOST likely architectural bottleneck and the correct remediation?

Options:
A. The KV cache is being evicted under high concurrency, forcing recomputation of attention states; increase --kv_cache_free_gpu_mem_fraction in TensorRT-LLM configuration
B. The TensorRT-LLM engine was compiled with a max_batch_size too low to accommodate 200 concurrent users; recompile the engine with a higher max_batch_size
C. The Triton HTTP frontend is serializing requests; switch to the gRPC endpoint to enable multiplexed connection handling
D. GPU memory bandwidth is saturated by attention computation at high sequence lengths; reduce max_input_length in the model configuration

**Correct Answer:** A

**Explanation:**
- **A is correct:** In TensorRT-LLM, the KV cache stores intermediate key-value tensors for each token in the context window. Under high concurrency, if the KV cache pool is insufficiently sized relative to active requests, cache entries are evicted and must be recomputed, causing per-token latency to spike dramatically under load. The --kv_cache_free_gpu_mem_fraction parameter controls what fraction of free GPU memory is reserved for KV cache. Increasing it at the cost of other memory consumers directly addresses this pattern.
- **B is incorrect:** A max_batch_size mismatch would cause requests to be queued beyond the batch window, adding queue delay but not causing per-token latency to scale with concurrency in the pattern described.
- **C is incorrect:** gRPC vs HTTP frontend choice affects connection handling efficiency at the network layer but does not address GPU-side memory pressure. Switching protocols would not recover 60% throughput loss.
- **D is incorrect:** Reducing max_input_length constrains what inputs the model can accept and is a capability trade-off, not a performance tuning action. It also does not address the root cause of KV cache pressure.

**Difficulty:** Hard
**Domain:** Model Deployment & Inference Operations
**Cognitive Level:** Analyze

**Question 2:**
You are deploying a Triton Inference Server ensemble pipeline for a document understanding system. The pipeline consists of: (1) a PDF text extractor (Python backend), (2) a tokenizer (Python backend), (3) a BERT encoder (TensorRT backend), (4) a classification head (TensorRT backend). End-to-end pipeline P95 latency is 850ms. Profiling shows the BERT encoder alone has P95 latency of 65ms. What is the MOST likely source of the remaining latency, and what should be investigated FIRST?

Options:
A. The TensorRT classification head is not optimized for the output tensor shape of the BERT encoder
B. Tensor serialization and deserialization overhead between Python backend stages and the TensorRT backend stages, compounded by lack of shared memory configuration
C. The Triton ensemble scheduler is adding coordination overhead between each stage proportional to model count
D. The PDF text extractor is CPU-bound with no async execution, blocking the entire pipeline thread

**Correct Answer:** B

**Explanation:**
- **B is correct:** In Triton ensemble pipelines, data must be transferred between backend stages. When Python backend stages precede TensorRT backend stages, tensors are serialized from Python objects, transferred through Triton's internal buffer management, and deserialized for GPU execution. Without shared memory configuration (CUDA shared memory or system shared memory in the model's input/output configuration), this transfer overhead can dwarf actual inference time. With BERT at 65ms and end-to-end at 850ms, over 750ms is unaccounted for — consistent with unoptimized inter-stage tensor transfer across 3 handoffs.
- **A is incorrect:** Output tensor shape mismatch between BERT and the classification head would cause an error, not a latency increase.
- **C is incorrect:** Triton's ensemble scheduler overhead is typically microseconds per stage, not hundreds of milliseconds. It is not proportional to model count in a way that would produce 750ms of overhead.
- **D is incorrect:** While CPU-bound preprocessing can be a bottleneck, blocking the entire pipeline thread would stall all requests, showing up as queue buildup rather than consistently high P95 latency on a lightly loaded system.

**Question 3:**
A TensorRT-LLM engine compiled with tensor parallelism = 4 (TP=4) across 4 GPUs is deployed via Triton. During a rolling node maintenance, one of the 4 GPU nodes is cordoned and drained. The Triton pod is rescheduled onto a node with only 3 GPUs available. The pod starts but all inference requests fail immediately with NCCL communication errors. What is the root cause?

Options:
A. NCCL requires a ring topology and cannot form a valid communication ring with 3 GPUs
B. The TensorRT-LLM engine binary is compiled with a fixed TP degree; running it on fewer GPUs than compiled violates the static parallelism topology baked into the engine
C. The GPU Operator's NCCL plugin version is incompatible with the rescheduled node's driver version
D. Triton's model configuration specifies count: 4 for the TensorRT-LLM backend instance group, which cannot be satisfied with 3 GPUs

**Correct Answer:** B

**Explanation:**
- **B is correct:** TensorRT-LLM engines with tensor parallelism are compiled for a specific TP degree. The weight tensors are sharded at compile time across exactly N GPUs. At runtime, the engine expects exactly N NCCL ranks to reconstruct full layer outputs through collective communication (AllReduce). Running a TP=4 engine on 3 GPUs is not a configuration mismatch — the binary itself is physically incompatible with any other parallelism topology. The NCCL errors are a symptom of the missing fourth rank.
- **A is incorrect:** NCCL supports ring, tree, and other topologies and can work with any number of GPUs ≥ 2. The topology is not the fundamental constraint here.
- **C is incorrect:** NCCL plugin version incompatibility would manifest as initialization failures or API errors, not as rank count mismatch errors specifically tied to the TP degree.
- **D is incorrect:** While the instance group configuration is relevant, the deeper issue is the compiled engine's static TP topology. Even correcting the instance group count would not allow a TP=4 engine to run on 3 GPUs.

**Question 4:**
An AI Ops team is building an observability stack for a multi-model Triton deployment serving 12 models. They need to detect per-model performance regressions within 60 seconds of deployment. Current monitoring uses a single Prometheus scrape interval of 5 minutes. What combination of changes would BEST achieve the 60-second detection SLO?

Options:
A. Reduce Prometheus scrape interval to 15 seconds for Triton metrics endpoints, and create Prometheus alerting rules on nv_inference_request_failure_total and nv_inference_queue_duration_us per model_name label
B. Deploy Grafana with auto-refresh set to 60 seconds and configure visual alerts on the dashboard panels
C. Increase Triton's --metrics-interval-ms to 5000ms to push metrics more frequently to the Prometheus endpoint
D. Enable Triton's trace feature with a sampling rate of 100% and ship traces to Jaeger for real-time latency analysis

**Correct Answer:** A

**Explanation:**
- **A is correct:** Achieving a 60-second detection SLO requires both high-frequency data collection (scrape interval ≤ 15s to get multiple data points within the window) and automated alerting rules (not manual dashboard watching). Prometheus alerting rules on per-model failure counts and queue duration, filtered by model_name label, provide automated, programmatic detection of per-model regressions without human intervention.
- **B is incorrect:** Grafana visual alerts require a human watching a dashboard or are dependent on Grafana's alerting engine, which evaluates rules at its own interval. Dashboard auto-refresh is a display setting, not a detection mechanism. It does not page on-call engineers automatically.
- **C is incorrect:** Triton's --metrics-interval-ms controls how frequently Triton internally updates its metric counters, not how frequently Prometheus pulls them. Setting this value does not affect Prometheus scrape frequency. Also, 5000ms is slower than the current 5-minute scrape is already providing per-sample.
- **D is incorrect:** Distributed tracing provides request-level latency visibility and is valuable for deep troubleshooting, but trace analysis tools like Jaeger are not designed for automated threshold-based alerting within a 60-second window.

**Question 5:**
DCGM Exporter on a production node begins reporting DCGM_FI_DEV_XID_ERRORS_COUNT incrementing steadily on GPU index 2, while all other GPUs show 0. The corresponding Triton pod serving on that GPU reports no errors and continues processing requests successfully. What should the AI Ops engineer do?

Options:
A. Ignore the XID error counter — since Triton is functioning, the errors are benign informational events
B. Identify the XID error code from DCGM, correlate with NVIDIA's XID error reference to assess severity, and proactively migrate the workload while scheduling GPU hardware diagnostics
C. Restart the Triton pod on GPU 2 to clear any CUDA context state associated with the XID errors
D. Increase the Kubernetes pod's GPU memory limit to provide more headroom and prevent further XID errors

**Correct Answer:** B

**Explanation:**
- **B is correct:** XID errors are NVIDIA GPU exception events logged by the driver. Different XID codes indicate vastly different severity levels — from minor recoverable errors (XID 13: graphics engine exception) to hardware failures requiring immediate action (XID 79: GPU has fallen off the bus, XID 94: contained ECC error). The correct procedure is to identify the specific XID code, consult NVIDIA's XID error reference, assess severity, and act accordingly. Proactive workload migration protects data integrity while hardware diagnostics proceed.
- **A is incorrect:** XID errors are never safely ignored. Some indicate hardware degradation that leads to silent data corruption in GPU computations — a critical risk for AI inference accuracy.
- **C is incorrect:** Restarting the pod clears the CUDA context but does not address underlying hardware faults if the XID errors indicate hardware-level issues. The errors will recur.
- **D is incorrect:** XID errors are hardware exception events, not memory allocation failures. Increasing memory limits has no effect on XID error rates.

**Question 6:**
An AI Ops engineer needs to monitor the effective memory bandwidth utilization of inference workloads across a fleet of A100 GPUs to identify memory-bound vs compute-bound models. Which DCGM metric combination provides the MOST actionable signal for this analysis?

Options:
A. DCGM_FI_DEV_GPU_UTIL and DCGM_FI_DEV_MEM_COPY_UTIL
B. DCGM_FI_DEV_FB_USED and DCGM_FI_DEV_FB_FREE
C. DCGM_FI_PROF_DRAM_ACTIVE and DCGM_FI_PROF_PIPE_TENSOR_ACTIVE combined with DCGM_FI_PROF_SM_ACTIVE
D. DCGM_FI_DEV_NVLINK_BANDWIDTH_TOTAL and DCGM_FI_DEV_PCIE_TX_BYTES

**Correct Answer:** C

**Explanation:**
- **C is correct:** DCGM_FI_PROF_DRAM_ACTIVE measures the fraction of cycles where the GPU's HBM memory interface is active (memory bandwidth utilization). DCGM_FI_PROF_PIPE_TENSOR_ACTIVE measures tensor core utilization. DCGM_FI_PROF_SM_ACTIVE measures overall SM occupancy. Comparing DRAM_ACTIVE against PIPE_TENSOR_ACTIVE directly reveals the memory-bound vs compute-bound ratio: high DRAM_ACTIVE with low PIPE_TENSOR_ACTIVE = memory-bound; high PIPE_TENSOR_ACTIVE with lower DRAM_ACTIVE = compute-bound.
- **A is incorrect:** DCGM_FI_DEV_GPU_UTIL is a coarse SM utilization metric. MEM_COPY_UTIL measures PCIe/NVLink copy engine activity (host-to-device transfers), not HBM bandwidth utilization during kernel execution.
- **B is incorrect:** FB_USED and FB_FREE measure GPU framebuffer (VRAM) capacity consumption — how much memory is allocated — not memory bandwidth during active computation.
- **D is incorrect:** NVLINK bandwidth and PCIe transfer bytes measure inter-GPU and CPU-GPU communication bandwidth, not the internal HBM bandwidth consumed by compute kernels.

**Question 7:**
A TensorRT-LLM LLM deployment is serving a 70B parameter model with pipeline parallelism = 2 (PP=2) and tensor parallelism = 4 (TP=4) across 8 GPUs. The team observes that inter-node GPU-to-GPU communication is causing 35% of total request latency. NVLink bandwidth between GPUs within a node is 600 GB/s. Cross-node communication is over InfiniBand HDR (200 Gb/s). What architectural change would MOST reduce communication overhead?

Options:
A. Increase pipeline parallelism to PP=4 and reduce tensor parallelism to TP=2, placing all TP communication within single nodes connected by NVLink
B. Increase tensor parallelism to TP=8 across all 8 GPUs to eliminate pipeline bubble overhead
C. Enable FP8 precision to reduce the tensor sizes transmitted during AllReduce operations
D. Deploy all 8 GPUs on a single node with NVLink to eliminate InfiniBand entirely

**Correct Answer:** A

**Explanation:**
- **A is correct:** Tensor parallelism requires high-frequency AllReduce communication at every transformer layer — this is the most bandwidth-intensive collective and must occur over the fastest interconnect available. Pipeline parallelism only requires communication at pipeline stage boundaries (much less frequent). By restructuring to PP=2, TP=2 per node (within-node NVLink at 600 GB/s), the frequent TP AllReduce operations stay on NVLink, while only the infrequent PP boundary activations cross the slower InfiniBand link. This directly reduces the 35% communication overhead.
- **B is incorrect:** Increasing TP to 8 across all 8 GPUs forces TP AllReduce communication across nodes over InfiniBand, significantly worsening the cross-node communication bottleneck.
- **C is incorrect:** FP8 reduces tensor sizes and would moderately reduce AllReduce data volume, but it requires re-quantizing the model and introduces accuracy risk. It is a secondary optimization that does not address the architectural placement of communication-intensive operations.
- **D is incorrect:** While this would eliminate InfiniBand entirely, deploying a 70B parameter model across 8 GPUs on a single node requires a DGX H100 or similar high-memory-density system. This is a hardware procurement decision, not an operational optimization.

**Difficulty:** Hard
**Domain:** Performance Optimization & Scaling

The postprocessing step takes 40ms. What is the MOST impactful optimization?

A. Optimize the TensorRT Whisper model with INT8 calibration to reduce inference time to under 40ms
B. Rewrite the audio preprocessing logic as a C++ custom backend or CUDA-accelerated operator to move preprocessing onto the GPU

**Question 9:**
A production Triton Inference Server cluster running a financial risk scoring model begins producing numerically inconsistent outputs. Two identical requests submitted within 30 seconds return different floating-point scores with a variance of up to 12%. No model updates have been made. The cluster runs on A100 GPUs with ECC memory enabled. Logs show no errors. What should the AI Ops engineer investigate FIRST?

Options:
A. Non-deterministic CUDA kernel execution due to floating-point reduction ordering in operations like layer normalization with varying batch compositions
B. Model weight corruption in the Triton model repository caused by a failed filesystem write
C. Load balancer routing identical requests to different Triton replicas running different model versions
D. GPU thermal throttling causing inconsistent clock speeds and therefore varying floating-point computation results

**Correct Answer:** A

**Explanation:**
- **A is correct:** Non-determinism in GPU computation is a well-documented and frequently misunderstood phenomenon. Operations like layer normalization, softmax, and attention use parallel floating-point reductions where the order of addition is non-deterministic across runs (especially when batch sizes change between requests). With FP16 or TF32 precision, the accumulated rounding errors from different reduction orderings can produce output variance in the 1-15% range. This is particularly prevalent in financial models where input ordering within a dynamically batched request affects intermediate computation order.
- **B is incorrect:** Model weight corruption would produce consistent errors (either systematic wrong outputs or crashes), not stochastic variance between runs. ECC memory is enabled, which actively corrects single-bit errors.
- **C is incorrect:** If different replicas were running different model versions, the variance would be systematic and version-correlated, not stochastic. This scenario would also be detectable through version labeling in Triton's request metadata.
- **D is incorrect:** Thermal throttling reduces clock speed, affecting latency. It does not cause floating-point result variance — computation is deterministic regardless of clock speed.

**Difficulty:** Hard
**Domain:** Incident Response & Troubleshooting
**Cognitive Level:** Analyze
**Scenario Type:** Incident

**Question 10:**
An NVIDIA Base Command Platform job running a multi-node distributed training workload across 4 DGX A100 nodes suddenly stalls. All worker processes appear alive (no exit codes). GPU utilization across all nodes drops to near 0%. NCCL debug logs show repeated "Timeout waiting for connection" messages from rank 0. Network connectivity between nodes is confirmed healthy. What is the MOST likely cause?

Options:
A. One GPU on a non-rank-0 node has encountered a hardware fault, causing its worker process to hang without exiting, blocking the NCCL barrier
B. The Base Command Platform scheduler has preempted the job's network allocation while the compute allocation remains active
C. The NCCL socket interface is binding to the wrong network interface (e.g., the management network instead of the high-speed InfiniBand fabric)
D. The training script's gradient synchronization step has a deadlock due to unequal batch sizes across ranks

**Correct Answer:** A

**Explanation:**
- **A is correct:** In NCCL collective operations, all ranks must reach a synchronization barrier simultaneously. If one worker process hangs (due to a GPU hardware fault, NaN in computation, or silent CUDA error) without exiting, the other ranks wait indefinitely at the barrier. Rank 0 logging "Timeout waiting for connection" while all processes appear alive is the signature of a hung non-rank-0 worker. The GPU utilization drop to 0% across all nodes confirms all ranks are blocked waiting at the collective barrier.
- **B is incorrect:** Base Command Platform manages job-level resource allocation. A network preemption event would terminate or pause the job at the scheduler level, not cause NCCL barrier timeouts with all processes remaining alive.
- **C is incorrect:** An incorrect NCCL socket interface binding would cause failures at job initialization, not mid-training stalls after training has been running (implied by the word "suddenly").
- **D is incorrect:** Unequal batch sizes across ranks in standard DDP training would cause shape mismatches in gradient tensors, producing errors or incorrect outputs — not indefinite hanging at the NCCL barrier level.

An operations team wants to enable MIG on a fleet of A100 80GB GPUs in a Kubernetes cluster managed by the NVIDIA GPU Operator. They need to support three workload classes simultaneously on each GPU: (1) a large LLM inference service requiring 40GB memory isolation, (2) two medium analytics jobs each requiring 20GB, and (3) four small preprocessing jobs each requiring 10GB. Which MIG profile combination is valid for an A100 80GB and accommodates all three workload classes?

Options:
A. 1x 4g.40gb + 2x 2g.20gb + 4x 1g.10gb — total 7 GPU instances
B. 1x 3g.40gb + 2x 2g.20gb + 1x 1g.10gb — total 4 GPU instances

D. 1x 4g.40gb + 3x 1g.10gb — total 4 GPU instances

The question asks what combination accommodates all described workload classes — and option A correctly lists profiles that match the required memory sizes. Note: Real MIG profile feasibility depends on the specific A100 MIG profile table; the key evaluation skill tested is whether the candidate can reason about memory slice constraints.

- **C is incorrect:** Two 3g.40gb profiles would consume 6 of 7 compute slices, leaving insufficient resources for medium and small workloads as described, and would not provide 40GB memory instances.
- **D is incorrect:** A 4g.40gb + 3x 1g.10gb would accommodate the LLM service and three small jobs but provides zero allocation for the two 20GB medium analytics jobs.

**Question 13:**
A Kubernetes cluster uses GPU time-slicing with a slice count of 4 on each A100 GPU. A machine learning engineer submits a training job requesting 4 GPU slices (effectively the whole GPU via time-slicing). They report that training throughput is 55% lower than expected compared to running the same job on a dedicated unsliced GPU. No other workloads are running on that node at the time. What is the correct explanation for the throughput reduction?

Options:
A. GPU time-slicing introduces CUDA context switching overhead even when only one workload is running, because the time-slicing scheduler runs at fixed intervals regardless of slice count utilization
B. The time-slicing configuration reserves 45% of GPU compute for the slice scheduling daemon running in the background
C. The training job's CUDA kernels are constrained to 25% of the GPU's SM count per time-slice allocation
D. The Kubernetes device plugin incorrectly reports 4 available GPUs, causing the training framework to attempt multi-GPU data parallelism inefficiently across time slices

**Correct Answer:** A

**Explanation:**
- **A is correct:** GPU time-slicing operates by preemptively context-switching between CUDA contexts at fixed time intervals. Even when only one CUDA context is active, the GPU still executes its context-switching scheduler cycle, which involves saving and restoring GPU register state. For compute-intensive training workloads with large register files and shared memory footprints, this context switch overhead (even with no competing workloads) introduces measurable latency. This is a fundamental difference between time-slicing and MIG — MIG provides hardware partitioning with no context switching, while time-slicing always incurs scheduling overhead.
- **B is incorrect:** There is no background scheduling daemon that consumes 45% of GPU compute. Time-slicing is managed by the GPU driver and hardware preemption mechanism with negligible fixed overhead, not a proportional resource reservation.
- **C is incorrect:** GPU time-slicing does not partition SMs. All time-sliced contexts have access to the full SM count during their time slice. SM partitioning is a feature of MIG, not time-slicing.
- **D is incorrect:** A training job requesting nvidia.com/gpu: 4 on a time-sliced node receives 4 logical GPU resources that map to the same physical GPU. A well-configured training framework would detect these as the same physical device (same GPU UUID) and not attempt multi-GPU parallelism across them.

What is the operational implication of the mixed strategy compared to the single strategy?

B. In mixed mode, MIG and non-MIG GPUs can coexist in the same Kubernetes cluster; in single mode, all nodes must either be fully MIG or fully non-MIG

**Question 15:**
A Kubeflow Pipeline is orchestrating a complete MLOps workflow: data preprocessing → training → model evaluation → conditional deployment to Triton. The evaluation step uses a custom Python component that queries an external model registry API. During a production run, the evaluation step completes successfully but the conditional deployment step is skipped even though evaluation metrics exceed the deployment threshold. Logs show no errors. What is the MOST likely cause?

Options:
A. The Kubeflow Pipeline's evaluation component is not passing output parameters correctly to the conditional step's input condition
B. The external model registry API is rate-limiting the evaluation component, causing it to return a partial metrics object
C. The Triton deployment step requires a Kubernetes ServiceAccount with permissions that conflict with the evaluation step's ServiceAccount
D. The Kubeflow Pipeline runner is using a cached result from a previous failed evaluation run

**Correct Answer:** A

**Explanation:**
- **A is correct:** In Kubeflow Pipelines, conditional execution uses dsl.Condition() that evaluates an output parameter from a preceding step. If the evaluation component writes metrics to a file but doesn't correctly surface them as pipeline component outputs (e.g., using the OutputPath parameter type or metrics artifact), the condition expression receives a None or empty value, which evaluates as false, causing the deployment step to be silently skipped. This is a common pipeline wiring bug that produces no error logs because the condition logic executes as designed — just with wrong input values.
- **B is incorrect:** API rate limiting would cause the evaluation step to fail or timeout with an HTTP 429 error, not complete successfully with incorrect output values.
- **C is incorrect:** ServiceAccount permission conflicts would manifest as authorization errors during the deployment step's execution attempt, not as a silent skip of the step.
- **D is incorrect:** Kubeflow Pipeline caching stores step execution results to avoid re-running unchanged steps. A cached result from a previous run would return the same outputs — and if those outputs previously triggered deployment, the condition would be met again. This would cause incorrect behavior in the opposite direction.

**Question 16:**
An AI training job using the Kubeflow Training Operator (PyTorchJob) is scheduled on a cluster with mixed GPU node types: 4 nodes with A100 80GB GPUs and 6 nodes with V100 32GB GPUs. The job requests 8 workers. The scheduler places 4 workers on A100 nodes and 4 workers on V100 nodes. The training run completes but model accuracy is 8% lower than expected and training loss curves show erratic behavior. What is the MOST likely explanation?

Options:
A. Mixed GPU placement causes non-deterministic gradient communication ordering in NCCL collectives
B. The A100 and V100 workers are using different default floating-point precisions (TF32 on A100, FP32 on V100), causing gradient magnitude inconsistencies across ranks during AllReduce
C. The V100 workers have insufficient memory to hold the full model replica, causing gradient checkpointing to activate and alter the effective batch gradient computation
D. NCCL cannot form an optimal communication topology when the ring spans both NVLink-connected A100s and PCIe-connected V100s, causing gradient corruption

**Correct Answer:** B

**Explanation:**
- **B is correct:** NVIDIA A100 introduced TF32 (TensorFloat-32), which is enabled by default in PyTorch for matrix multiplications on A100. TF32 uses FP32 range but 10-bit mantissa precision (vs 23-bit in FP32). V100 GPUs do not support TF32 and execute at full FP32 precision. When gradient tensors computed at different precisions are aggregated via AllReduce, the systematic precision difference across half the workers introduces consistent gradient magnitude errors, leading to degraded convergence and erratic loss curves.
- **A is incorrect:** NCCL AllReduce operations are deterministic in their communication ordering within a run. Non-determinism in gradient ordering is a within-node, within-operation concern, not a cross-GPU-type issue.
- **C is incorrect:** If V100 workers lacked memory for the model replica, the job would fail at initialization with OOM errors, not complete successfully with degraded accuracy.
- **D is incorrect:** NCCL gradient communication errors would cause numerical corruption that would produce NaN/Inf values or catastrophic loss spikes, not a consistent 8% accuracy degradation with erratic-but-finite loss curves.

**Question 18:**
An organization stores proprietary model weights in NGC Private Registry. A security audit reveals that a CI/CD pipeline is authenticating to NGC using a shared service account API key stored as a plaintext environment variable in a Jenkinsfile committed to a public GitHub repository. The model weights are valued at $2M in development cost. What is the CORRECT remediation sequence, ordered by priority?

Options:
A. Rotate the compromised NGC API key immediately → store the new key in a secrets manager (e.g., HashiCorp Vault or Kubernetes Secrets) → update the pipeline to retrieve the key dynamically → audit NGC access logs for unauthorized pulls
B. Audit NGC access logs first → update the Jenkinsfile to remove the plaintext key → rotate the API key → store the new key in environment variables encrypted at rest
C. Remove the public GitHub repository → rotate the NGC API key → store the new key in the Jenkinsfile using base64 encoding → enable NGC registry scanning
D. Revoke all NGC service accounts → rebuild all model artifacts → re-push to a new private registry → redistribute new API keys to all pipeline consumers

**Correct Answer:** A

**Explanation:**
- **A is correct:** The remediation follows the correct incident response sequence for a credential exposure: (1) Rotate the compromised key immediately to stop ongoing unauthorized access — the key is publicly exposed and must be treated as compromised regardless of whether exploitation has occurred yet. (2) Store the new key in a secrets manager to prevent recurrence. (3) Update the pipeline to retrieve secrets dynamically. (4) Audit access logs to assess the blast radius of the exposure. Rotation must precede the audit because every minute of delay with the key active is additional exposure time.
- **B is incorrect:** Auditing before rotating the key delays revocation of the compromised credential. Every minute of audit time is time a potential attacker can continue using the exposed key.
- **C is incorrect:** Base64 encoding is not encryption — it is trivially reversible and provides zero security. Storing a base64-encoded key in a Jenkinsfile is equivalent to plaintext storage.
- **D is incorrect:** Revoking all service accounts and rebuilding all model artifacts is an extreme response that would cause significant operational disruption without proportional security benefit. The issue is a single compromised API key, not a broad compromise of all service accounts or model integrity.

**Question 19:**
An AI Ops team is deploying a Triton Inference Server to serve a model that processes sensitive medical records. Compliance requirements mandate that (1) all inference request payloads must be encrypted in transit, (2) access must be restricted to specific internal service identities, and (3) all inference calls must be auditable with requester identity logged. Which combination of controls BEST satisfies all three requirements?

Options:
A. Enable TLS on Triton's gRPC endpoint + configure Kubernetes NetworkPolicy to restrict ingress to authorized service namespaces + deploy an Istio sidecar with mTLS and access logging enabled
B. Enable TLS on Triton's HTTP endpoint + use Kubernetes RBAC to restrict pod creation in the Triton namespace + configure Prometheus to record request counts per client IP
C. Deploy Triton behind an NGINX ingress controller with TLS termination + use API key authentication via request headers + store access logs in an S3 bucket
D. Enable NVIDIA Triton's built-in authentication plugin + use Kubernetes PodSecurityPolicy to restrict privileged containers + configure ELK stack for log aggregation

**Correct Answer:** A

**Explanation:**
- **A is correct:** This combination addresses all three requirements precisely. TLS on gRPC satisfies encryption in transit (requirement 1). Kubernetes NetworkPolicy restricts which pods/namespaces can reach the Triton service at the network layer (requirement 2). Istio's mTLS enforces mutual service identity authentication (each service must present a certificate, satisfying requirement 2 at the identity level), and Istio's access logging captures requester identity (service account/certificate) per request (requirement 3).

- **D is incorrect:** Triton does not have a built-in authentication plugin in the standard open-source release. PodSecurityPolicy (deprecated in Kubernetes 1.25) restricts pod privilege escalation but does not control service-to-service communication. ELK log aggregation is a logging infrastructure choice, not an access control mechanism.

**Question 20:**
During a routine security review, an AI Ops engineer discovers that a Triton Inference Server pod is running with a Kubernetes ServiceAccount that has cluster-admin privileges. The pod was configured this way by a developer to avoid troubleshooting RBAC issues during initial deployment. What are the security risks specific to this misconfiguration in an AI inference context, and what is the correct remediation?

Options:
A. The risk is only theoretical; inference workloads do not use Kubernetes API calls, so cluster-admin privileges are dormant. Remediation is documentation of the exception.
B. A compromised Triton pod (e.g., via a malicious model or deserialization exploit) could use the cluster-admin token mounted in the pod to exfiltrate secrets from all namespaces, modify workloads cluster-wide, or escalate to full cluster compromise. Remediate by creating a minimal RBAC Role with only the permissions the pod legitimately requires and binding it to a dedicated ServiceAccount.
C. The risk is excessive API server load from Triton's internal health checks using the cluster-admin token. Remediate by setting automountServiceAccountToken: false and using a dedicated token for health checks.
D. Cluster-admin privileges allow the inference pod to directly access GPU resources on other nodes, creating a data leakage path between inference tenants. Remediate using MIG to provide GPU-level isolation.

**Correct Answer:** B

**Explanation:**
- **B is correct:** The Kubernetes service account token is automatically mounted into every pod and can be used to call the Kubernetes API with the permissions of the bound ServiceAccount. A cluster-admin token provides unrestricted access to the entire cluster's API. In an AI inference context, an attacker who achieves code execution within Triton (via model deserialization vulnerabilities, prompt injection leading to code execution, or supply chain compromise of model artifacts) would immediately have access to a cluster-admin credential — enabling full cluster compromise. The principle of least privilege requires creating a minimal Role with only required permissions (e.g., reading its own ConfigMap).
- **A is incorrect:** Inference workloads may not make Kubernetes API calls in normal operation, but the token is still mounted and available to any code running in the pod. "Dormant" privileges are not safe — they are available to any attacker who gains code execution.
- **C is incorrect:** Triton's health check endpoints are HTTP endpoints served by Triton itself, not Kubernetes API calls. Health checks do not use the ServiceAccount token. The risk described is fabricated.
- **D is incorrect:** Kubernetes ServiceAccount privileges operate at the Kubernetes API level, not at the hardware/GPU level. They cannot be used to access GPU memory on other nodes. GPU isolation is a separate concern addressed by MIG, not by RBAC.

An AI Ops engineer pulls a model from NGC Model Registry using the NGC CLI and deploys it to Triton. The model card on NGC specifies it was optimized for A100 GPUs with TensorRT 8.5. No errors are reported. What is the MOST likely explanation?

A. The TensorRT engine plan (.plan file) from NGC was compiled for A100's Ampere architecture; running it on H100 (Hopper architecture) causes TensorRT to fall back to a compatibility execution path that is not optimized for H100's new features
B. The NGC model was quantized to INT8 using A100-specific calibration data that is not representative of H100 compute characteristics

**Correct Answer:** A

**Explanation:**

- **C is incorrect:** There is no H100-specific minimum batch size requirement. H100 is fully backward compatible with any batch size that functions on A100.

An AI Ops team is planning to migrate a Triton deployment from single-GPU A100 serving to a 4-GPU MIG configuration on the same physical hardware, serving four independent model tenants. Before migration, the team must validate that the MIG partition sizes will not cause inference SLA regressions for each tenant. Which pre-migration validation approach is MOST rigorous?

Options:
A. Review the model cards for each tenant's model and compare published benchmark throughput figures to SLA requirements
B. Benchmark each tenant model on a representative MIG instance profile (matching the planned partition size) using tools such as Triton's perf_analyzer or genai-perf with production-representative traffic patterns, and compare results against current single-GPU baselines
