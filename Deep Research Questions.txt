Question 1: A Triton Inference Server is deployed without dynamic batching. The metrics show: Request Count = 2, Inference Count = 9, Execution Count = 2. What does the Execution Count of 2 indicate?

Options:
A. Dynamic batching is enabled and batched the requests into two executions.
B. Dynamic batching is disabled; each request executed separately. (Correct)
C. The server is using two GPUs in parallel.
D. The server is running in synchronous mode, limiting executions to two.

Correct Answer: B

Explanation: In Triton metrics, Execution Count = 2 for two requests means each request was executed separately (no dynamic batching)
. If dynamic batching were enabled, the two requests could have been combined into one execution (Execution Count = 1)
. Options A, C, and D are incorrect because the log indicates no batching (two executions) and it’s not related to multi-GPU or synchronous vs asynchronous modes.

Question 2: An AI cluster uses NVIDIA A100 GPUs in multi-tenant mode. Currently, GPUs are allocated to jobs exclusively, and utilization is low. What change will allow secure sharing of GPU memory and compute among multiple workloads?

Options:
A. Enable Multi-Instance GPU (MIG) mode on the A100 GPUs. (Correct)
B. Enable NVIDIA Multi-Process Service (MPS) on each GPU.
C. Switch to NVIDIA GRID vGPU licensing.
D. Install NVIDIA Virtual GPU Manager.

Correct Answer: A

Explanation: MIG (Multi-Instance GPU) partitions Ampere-class GPUs (like A100/H100) into multiple independent GPU instances, each with dedicated memory and compute
. This allows multiple tenants to share a single GPU securely. MPS (option B) allows CUDA contexts to share a GPU but does not isolate memory. Options C and D refer to vGPU (for virtual desktops) which is not applicable to data-center training. The correct answer is MIG
.

Question 3: A new Kubernetes node with NVIDIA GPUs is added, but pods requesting nvidia.com/gpu resources remain pending. The node has the NVIDIA GPU driver installed. What is the most likely cause?

Options:
A. The NVIDIA device plugin is not installed on the cluster. (Correct)
B. The node’s CPU has insufficient cores for GPU scheduling.
C. The MIG Manager is enabled, preventing standard GPU scheduling.
D. The node lacks an Ethernet connection to the network.

Correct Answer: A

Explanation: In Kubernetes, the NVIDIA device plugin (installed via GPU Operator or daemonset) advertises the nvidia.com/gpu resource on GPU nodes. If it’s missing, pods will remain pending even if the driver is present. Options B and D are irrelevant to GPU resource advertising. Option C is incorrect because MIG Manager would show MIG node labels but wouldn’t block GPU scheduling entirely. The lack of the device plugin is the common cause. (No official doc cited here.)

Question 5: In a Kubernetes AI cluster, you need to collect real-time GPU health and utilization metrics from all nodes. Which NVIDIA tool is designed for this purpose?

Options:
A. nvidia-smi command on each node.
B. NVIDIA DCGM with the DCGM Exporter. (Correct)
C. NVIDIA Management Library (NVML) directly.
D. NVIDIA CUDA Profiler.

Correct Answer: B

Explanation: NVIDIA DCGM (Data Center GPU Manager) integrates with Kubernetes via a Prometheus exporter to gather GPU telemetry across the cluster
. This includes health monitoring and utilization. The nvidia-smi CLI (option A) only works per node, not cluster-wide, and NVML (option C) is a library interface, not a deployed monitoring solution. CUDA Profiler (D) is for profiling kernels, not cluster monitoring. DCGM-Exporter is the correct choice
.

Question 6: An A100 GPU node is set up with 7 MIG instances (1g.5gb profile) per GPU, but a pod scheduled to one instance still runs out of memory. What should be adjusted to allow sufficient GPU memory for that job?

Options:
A. Switch to fewer, larger MIG instances on the GPU. (Correct)
B. Increase the node’s system RAM.
C. Disable MIG so the pod can use the full GPU.
D. Enable NVIDIA Multi-Process Service (MPS).

Correct Answer: A

Explanation: If a single pod is OOM-killing despite MIG instances, it likely needs more GPU memory. Configuring fewer, larger MIG profiles (e.g. 2g or 3g profiles) gives each instance more memory. Disabling MIG (C) would lock the pod to one GPU but breaks multi-tenancy. Increasing host RAM (B) doesn’t affect GPU memory. MPS (D) won’t allocate more GPU memory. The right approach is to reconfigure MIG with larger instance profiles so the workload has more memory.

Question 9: A batch inference job on Kubernetes requests 8 GPUs but none are allocated. The kubectl describe pod shows “Insufficient gpu”. GPU nodes exist. What likely misconfiguration causes this?

Options:
A. The pod’s resource request should use nvidia.com/gpu instead of a generic GPU resource.
B. The GPU Operator’s NVIDIA driver must be manually installed on each node.
C. The MIG mode must be disabled for inference workloads.
D. The node must have a taint that matches the pod’s toleration. (Correct)

Correct Answer: D

Explanation: An “Insufficient gpu” error despite available GPUs often means the pod’s scheduling constraints don’t match the node. If the nodes are tainted for GPUs (e.g. nvidia.com/gpu=present:NoSchedule) but the pod lacks a matching toleration, it won’t schedule. Option A is incorrect because nvidia.com/gpu is the correct resource name. Option B is misleading – the GPU Operator already handles driver installation. MIG (C) would restrict GPUs, not completely block scheduling. The most realistic cause is a Kubernetes taint/toleration issue, so D is correct.

Question 12: A DGX system with NVSwitches reports repeated fatal GPU errors (SXid) in dmesg. The NVIDIA Fabric Manager service is currently disabled. What is the most effective way to resolve the NVSwitch errors?

Options:
A. Enable and start the NVIDIA Fabric Manager service on the node. (Correct)
B. Disable NVSwitch mode in the GPU BIOS.
C. Uninstall the GPU drivers and reinstall them.
D. Move GPUs to another PCIe slot.

Correct Answer: A

Explanation: SXid errors on NVSwitch-connected GPUs typically indicate the fabric manager (FM) is not running or configured. NVIDIA Fabric Manager must run (usually as root) to manage the NVSwitch fabrics
. Enabling FM will configure the NVSwitch topology and resolve SXid issues. Disabling NVSwitch (B) is not a practical option, and driver reinstall (C) or moving slots (D) will not address the fabric configuration. The FM documentation notes FM must run as root to configure NVLinks/NVSwitches
.

Question 15: In an NVIDIA Run:AI-managed Kubernetes cluster, project A has higher rank than project B within the same department and node pool. Both have running GPUs. Project C (different department) has exhausted its quota and is over-subscribed. Which workloads will the scheduler consider preempting first?

Options:
A. Any lower-priority workload in a higher-ranked project.
B. Workloads based on cross-project fairness, not fixed priority (correct).
C. Any workload in the lowest-ranked department, regardless of project priority.
D. The scheduler does not preempt in this situation.

Correct Answer: B

Explanation: NVIDIA Run:AI scheduling uses a fairshare approach across projects and departments
. It only uses workload priority/preemption within a project, and cross-project or cross-department choices are based on fairness, not fixed priorities
. Option A is incorrect (priorities apply only within a queue). Option C is also wrong – it’s fairshare, not simply by department rank. Option D is incorrect since fairness-based preemption can occur when oversubscribed. Thus, B is correct.

Question 16: A department’s project is allowed 8 GPUs, and another project is allowed 4 GPUs, but only 9 GPUs are physically available. Both projects are running in-quota and have pending workloads. According to NVIDIA Run:AI’s scheduling policy, how are the leftover GPUs distributed?

Options:
A. The extra 1 GPU is split as 0.5 GPU to each (impossible).
B. The extra GPU goes entirely to the department with the higher rank project.
C. Over-quota resources (the 1 GPU) are first pooled at the department level, then shared by project rank/fairness
. (Correct)
D. All projects must drop to fit within 9 GPUs; no over-allocation is allowed.

Correct Answer: C

Explanation: Run:AI uses a multi-level quota and fairshare system. Each department’s deserved (quota) resources are filled first, then any excess (over-quota) GPUs are pooled at the department level and allocated to projects based on rank/fairness
. Thus, the 1 extra GPU is shared according to those rules. Option B is too simplistic, and D is false – Run:AI does allow over-quota scheduling. Option C correctly describes the hierarchy of quota -> department -> project for over-quota resources
.
